# Knowledge Keeper — 方法论（万金油）

> 让任意项目的 AI ①精通该项目细节 ②随代码自动保鲜知识。
> 从一个生产级 Java 项目实战跑通后抽象而来，非纸上谈兵。

## 0. 它解决什么
让 AI 不再"每次重读代码、且常凭记忆猜"，而是有：
- **skills**(程序性，自动触发)：本项目"怎么做某类活"。
- **knowledge**(参考，按需读)：本项目"是什么/为什么/有哪些坑"。
- **自迭代闭环**：代码一变 → 确定性脚本算出哪篇文档过时 → 会话结束极简提醒 → 维护技能核实代码后同步。不靠 AI"记得更新"。

## 1. 两条铁律（决定成败）
1. **分层**：skills=怎么做，knowledge=是什么/为什么。别混。
2. **准确性纪律**：只基于实际读到的代码写，禁止凭记忆/猜测；关键断言标 `文件:行号`；拿不准标 `⚠️未验证`；宁缺毋编。违反一次，整库可信度就崩。

## 2. 机制怎么来（插件 vs 手动）
- **用了 knowledge-keeper 插件**(推荐)：漂移检测脚本 + Stop hook + `skill-maintenance` 维护技能**插件已自带**，启用即生效。你/AI 只需产出**内容**(knowledge + skills 文档)。
- **不用插件、手动装**：把本插件 `scripts/` 下两个脚本拷进项目 `scripts/`，把 Stop hook 写进项目 `.claude/settings.json`，再忽略 `.claude/.drift-state`。(详见本插件 README"手动装"一节。)

## 3. 锚定 frontmatter（自迭代地基）
每篇受漂移管理的文档头部：
```yaml
---
name: <kebab-case>
description: <做什么 / 何时读或触发；skill 的 description 要略 pushy>
sources:                 # 只列推导来源的代码路径(相对仓库根)
  - path/to/foo/**
verified_at: <短 git SHA>
---
```
- `sources` 窄而准。无 `sources`/`verified_at` 的文档(如 index.md)不参与检测。

## 4. 执行流程（建库）
- **Phase 1 测绘**：模块结构、规模、`git log` 活跃区、分层入口；复用已有文档；产出"域→源码路径"表给用户确认。
- **Phase 2 多 agent 抽取 + 对抗式审查**：每域一个 agent 读真实代码写代码级文档(带锚定+file:line)；每篇再派"怀疑论者"核对断言、核不实即修。顺带挖到的真实 bug 只记进 landmines、不夹带改代码，单独汇报。
- **Phase 3 组织**：knowledge 每域一篇；skills 按**高频开发动作**选题(非按模块!);`landmines.md` 汇总坑/bug/陷阱；建 `index.md`。
- **Phase 4 (可选) GAP 配置**：建 `.claude/knowledge-drift.config`(每行一个业务模块 glob)启用覆盖缺口检查。
- **Phase 5 维护循环**：提交影响锚定源码的改动后，AI 自主用 `skill-maintenance` 同步(只做必要改动、不逐条问用户)；Stop hook 当跨会话安全网(同一漂移只提醒一次)；用户纠正=最高优先同步时机。

## 5. 原理 / 为什么这么设计
| 朴素做法 | 本方案 | 差异 |
|---|---|---|
| hook 提醒"你动了 src/" | 文档锚定 sources+verified_at，脚本算 `git diff ∩ sources` | 精确到"哪篇因哪文件过时" |
| 靠 AI 记得更新 | 确定性脚本检测，无漂移静默(0 token) | 不靠记忆、不烧 token |
| 检测即改 | 检测(脚本)与修改(技能)分离 | 安全，避免误改 |

- **不做"代码一改就自动改文档"**：烧 token、频繁触发、误改风险。检测自动且廉价，修改必须经判断。
- **减噪 + 防循环**：指纹去重(同一组漂移只提醒一次) + `stop_hook_active` 兜底。

## 6. 诚实边界 / 坑
- 脚本依赖 `python3` + `git`(无第三方包)。
- `sources` 过宽=噪音、过窄=漏报——宁可稍窄，漏报由"用户纠正"和周期审计兜底。
- 脚本只发现"源码动了"，发现不了"语义变了但文件没动/动了别处"——"用户纠正"这条人工触发不可省。
- **sources 两条铁律**：① 只列代码、绝不列另一篇文档(否则那篇一改连带误报)；② 慎列被很多文档当示例的"热点宽文件"(它一改触发一批 STALE)。
