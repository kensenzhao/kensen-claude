# skill-maintenance — 知识体系自迭代

把"代码变了 → 文档过时"闭环。检测由 knowledge-keeper 插件的漂移脚本自动做(确定性、零成本)，**修改由本技能经判断完成**(可追溯、不乱改)。

## 何时触发

| 触发源 | 信号 |
|--------|------|
| **我改了业务代码**(主力) | 我提交了影响已锚定源码的改动 —— 提交后主动核对、自主同步,只做必要改动、不逐条征求用户 |
| **用户纠正**(最高优先) | 用户说"不对,本项目约定是 X""这里逻辑其实是 Y" —— 立即固化进对应文档 |
| **Stop hook**(安全网) | 会话结束时插件 hook **同一组漂移只极简提醒一次**,提示可能有未同步文档 |
| **覆盖缺口** | 漂移脚本报 GAP：某业务模块无任何知识文档锚定 |

**工作方式**：自主谨慎决策,不是逐条问用户。检测自动且廉价;是否同步、改哪些由你判断,只做必要改动;仅业务语义抉择(是 bug 还是有意为之)才向用户确认。

## 铁律

1. **只基于实际读到的代码写**,禁止凭记忆/猜测。改文档前先 Read 变化后的真实源码。
2. 关键断言标 `相对路径:行号`。
3. 拿不准标 `⚠️未验证`,宁缺毋编。
4. 改完**必须**把该文档 `verified_at` 推到当前 HEAD(`git rev-parse --short HEAD`),否则漂移脚本会一直报它。
5. 改文档**不顺带改代码**;发现代码 bug 单独告知用户。

## 标准流程

```
1. 定位受影响文档
   - 漂移触发: 看 Stop hook 的提示;或重跑插件的 check-knowledge-drift.py
     (没有现成入口时,可手动复现其逻辑: git diff <verified_at>..HEAD ∩ 文档 sources)
   - 用户纠正: 从纠正内容判断属于哪个域 → 对应文档
2. 读真实代码: 打开变更文件 + 文档 sources 指向的源码,核对当前事实
3. 更新内容: 修正失真断言、补全新增逻辑、更新 file:line;sources 有新增覆盖路径一并补
4. 推进锚点: verified_at 改为当前 HEAD 短 SHA
5. 处理 GAP: 为未覆盖模块新建文档(套模板+锚定),并在 index.md 加指针
6. 复跑确认: 漂移检测应清零;仍报则回步骤 2
```

## 反模式
- ❌ 无脑重写整篇(只改受影响部分)。
- ❌ 改完忘推 `verified_at`(下次又被报,狼来了)。
- ❌ `sources` 写过宽 / 列了另一篇文档(噪音)。
- ❌ 没读代码、凭印象改(破坏可信度)。

完整方法论见本插件 `reference/playbook.md`。
